openclaw.json 里，和 Bot 最相关的 key 深挖 🧬

这一页只讲你最可能动到、也最容易改错的 bot 相关配置，不再平均铺开所有 top-level key。目标是让你以后看到某个多 bot 问题时，能立刻定位到该查哪一行。

🎯 先给结论，Bot 相关最关键的 4 组 key

  1. agents.list[]，定义有哪些 bot 对应的 agent
  2. bindings[]，定义哪个账号路由到哪个 agent
  3. channels.discord.defaultAccount，定义默认 Discord 账号
  4. channels.discord.accounts.<id>，定义有哪些 Discord bot 账号实例

1️⃣ agents.list[] 每个字段到底有什么用

idagent 唯一标识。
这是整套多 bot 系统里的内部主键。binding、session、memory、运行状态，最终都围绕它组织。
workspace这个 agent 的工作区。
如果你发现 bot 风格不对、记忆不对、规则不对，先看这里指向哪个目录。
thinkingDefault默认思考强度。
不决定人格，但会影响“它愿意花多少脑力”。适合让轻 bot 更轻，重 bot 更稳。
identity.name显示名。更多是展示层，不是完整人格。
identity.emoji快速识别 bot 的视觉符号。
identity.avatar头像资源路径。通常和 bot workspace 里的 avatar 资源配合。
identity.theme主题标签，偏展示和语义辅助。
default默认 agent。
它更像系统默认脑子，不等于某个渠道一定默认用它。渠道最终还得看 binding 命中。

2️⃣ bindings[] 是真正的路由总开关

agentId这条匹配命中后，消息送给哪个 agent。
match.channel来自哪个渠道，比如 discord、telegram、whatsapp。
match.accountId来自哪个账号实例。多 Discord bot 时基本都靠这个分流。
match.peer更细粒度条件，比如只匹配某个群/用户/thread。你现在还没主要用这层，但以后很有用。
💡 一个常见误区是：以为 bot 风格串了是 workspace 问题。实际上，先排 binding 更常见。因为只要 accountId 绑错，再漂亮的人格文件也会读错人。

3️⃣ channels.discord.defaultAccount 到底重要在哪

它定义的是：当某些地方没有显式指定 Discord accountId 时，优先用哪个账号。你现在是 default。

  • 它不是默认 agent
  • 它不是默认人格
  • 它更像“默认出口账号”
如果以后你扩更多 bot，默认账号最好始终保持语义清楚。否则排查时会很绕。

4️⃣ channels.discord.accounts.<id> 是账号实例层，不是人格层

<id>账号实例名，比如 trading_report_bot。
token该 Discord bot 的登录 token。
它只负责“这个账号能不能接入 Discord”，不负责这个 bot 像谁。
🔐 最容易搞混的一点：账号接入成功 ≠ 路由正确 ≠ 人格正确。这是三层不同问题。

🧪 三种典型故障，你该查哪

Bot 完全不回先查 channels.discord.accounts.<id>.token、enabled、groupPolicy、guild allowlist。
Bot 回了，但像错人先查 bindings[] 和 agents.list[].workspace。
Bot 人设对，但能力没分开先查 agents.defaults.skills / agents.list[].skills。
