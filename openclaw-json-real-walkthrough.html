你当前真实 openclaw.json 的逐段解读 🔍

这一页不是抽象语法，而是对你当前机器里这份 ~/.openclaw/openclaw.json 的结构做解释。为了安全，不直接贴出 secret，但会按真实字段组合讲它在你系统里实际意味着什么。

① meta + wizard

这两块主要是系统自留痕迹。

  • meta.lastTouchedVersion / lastTouchedAt 表示最近一次被哪个版本、什么时候碰过。
  • wizard.* 表示最近一次跑过 configure 向导的情况。

结论：这两块对 bot 日常行为几乎没直接影响，更多是运维排查信息。

② browser

你这里启用了浏览器，并且明确指定了系统 Chrome 路径，还加了 --headless=new、--disable-gpu 等参数。

这意味着：

  • 你的 OpenClaw 默认具备浏览器自动化能力
  • 运行环境更偏服务器/WSL 风格，所以用 headless
  • noSandbox: true 也说明这台机子的浏览器运行更偏“能跑起来优先”

③ auth

你这里的 auth profiles 里有多个 OpenAI / OpenAI Codex 档案。

实战理解：

  • 这是“模型认证层”
  • 不是 bot 路由层
  • 不是人格层

所以如果某个 bot 回答风格不对，先别怀疑 auth。auth 更像“这些脑子靠什么登录模型服务”。

④ agents.defaults

这是你系统的公共基线。你现在的公共基线很清楚：

  • 默认主模型：openai-codex/gpt-5.4
  • 有 fallback 模型
  • 默认 workspace：~/.openclaw/workspace
  • 默认 thinking：medium
  • 默认 memory search：本地

它的意义：不给每个 agent 重复写一遍通用设置，让 bot 在这条公共底线上分化。

⑤ agents.list，这里是你的多 Bot 主体

你当前定义了 6 个 agent：

  • main
  • trading_report_bot
  • family_health_bot
  • library_growth_bot
  • info_collection_bot
  • skills_master_bot

更关键的是：这 6 个 agent 各自都指向不同 workspace，所以它们不是“同一个脑子换头像”，而是真正分工了。

  • main 用主 workspace
  • 其余 5 个都用 runtime-adapter/bot-workspaces/<bot> 里的独立工位

⑥ bindings，这里是多 Bot 能跑顺的关键

你现在的 bindings 很干净：

  • trading_report_bot 账号 -> trading_report_bot agent
  • family_health_bot 账号 -> family_health_bot agent
  • library_growth_bot 账号 -> library_growth_bot agent
  • info_collection_bot 账号 -> info_collection_bot agent
  • skills_master_bot 账号 -> skills_master_bot agent

默认 Discord 账号 default 没单独写 binding，但实际就走 main。

💡 这套结构的好处是：排查特别直接。你看到 bot 账号，就几乎能立刻想到它落到哪个 agent。

⑦ tools + commands + session

  • tools.profile = coding，说明这套系统整体更偏可执行、可开发、可操作，而不只是聊天。
  • tools.web.search 开着，provider 是 DuckDuckGo。
  • tools.elevated 也开着，但 allowFrom 做了来源限制。
  • commands.native / nativeSkills 是 auto，说明系统倾向自动接原生命令与技能命令。
  • session.dmScope = per-channel-peer，说明 DM 会话按“渠道 + 对端”切。

这几块的合起来的效果：你的 OpenClaw 更像一个带工具执行力的多 Agent 系统，而不是只有文字回复的 bot。

⑧ channels.discord，这里是 6 个 bot 账号入口层

你当前 Discord 配置体现出三件事：

  1. Discord 已启用
  2. 默认账号是 default
  3. 下面挂了 6 个账号实例

最重要的事实不是 token 数量，而是：你已经把多账号结构搭起来了，而且能和 agent 层对上。

⑨ gateway

  • 端口是 18789
  • 模式是 local
  • bind 是 loopback
  • gateway 自己有 token 鉴权
  • tailscale 目前关闭
  • 对 nodes 额外禁了一些高风险命令

结论：这是一套偏本机私有运行、默认不向外裸露的配置风格，安全感比“全开放”高很多。

⚠️ 看真实配置时，最容易犯的错是把“账号层”“路由层”“人格层”“能力层”混在一起。你现在这套系统其实已经拆得不错了，后面更多是精修，而不是重构。
